# Object-Level Matching: What It Takes to Link a Unit

This document explains what "Matching" means in `objects.json` at the COFF object level — what sections must match, what the linker needs, and what gaps are acceptable.

## Background: Function Matching vs Object Matching

The orchestrator database tracks **function-level** matching: each function in `.text` gets a match percentage from objdiff. But marking a unit as `"Matching"` in `objects.json` is an **object-level** claim. It means: "this compiled `.obj` can replace the split `.obj` in the final linked binary without breaking anything."

A unit where every function is 100% matched in `.text` is *not* automatically ready to link. The COFF object contains multiple sections, and the linker consumes all of them.

## COFF Sections in a Typical Unit

When MSVC compiles a `.cpp` file with `/O1 /GR /EHsc`, the resulting `.obj` contains:

| Section | Content | Matching Requirement |
|---------|---------|---------------------|
| `.text` | Function machine code | Must match instruction-by-instruction (the main decomp work) |
| `.rdata` | Vtables, RTTI type descriptors, string literals, float constants, `const` globals | Must have identical symbols with correct content |
| `.data` | Initialized global/static variables | Must have correct sizes and initial values |
| `.bss` | Uninitialized global/static variables | Must have correct sizes |
| `.pdata` | Exception table (`IMAGE_CE_RUNTIME_FUNCTION_ENTRY` array) | Generated by compiler from function structure — matches if `.text` matches |
| `.xdata` | Exception handler unwind data (rare) | Generated by compiler for `try`/`catch` blocks |
| `.text$x` | `__unwind$` exception cleanup thunks (COMDAT) | Generated by compiler for EH-enabled functions |
| `.text$yc` | RTTI `complete_object_locator` data | Compiler-generated, depends on class hierarchy |
| `.text$yd` | Dynamic initializer/destructor thunks (`??__E`/`??__F`) | Compiler-generated for static objects |

### What objdiff Actually Compares

objdiff compares the split `.obj` (extracted from XEX) against the compiled `.obj` (from decomp source), section by section:

- **`.text`**: Instruction-level fuzzy match per function. This is the `fuzzy_match_percent`.
- **`.pdata`**: Byte-level match of exception table entries. Usually 100% if `.text` functions match.
- **`.rdata`**: Data-level match. Often lower than `.text` because of address-dependent content (vtable entries contain absolute VAs that differ in our build).
- **`.data`/`.bss`**: Size and content match for globals.

The unit-level `fuzzy_match_percent` only reflects `.text`. A unit can show 100% `.text` match while `.rdata` is at 18% — this happens when vtable entries or RTTI contain baked-in addresses from the original binary.

## The Linking Pipeline

When `objects.json` marks a unit as `"Matching"`, `configure.py` links **both** the decomp `.obj` and the split `.obj`:

```
Matching unit:     link(decomp.obj + split.obj)   # /FORCE:MULTIPLE picks first definition
NonMatching unit:  link(split.obj only)            # original code, guaranteed correct
```

The linker uses `/FORCE:MULTIPLE` to resolve duplicate symbol definitions — it takes the first one encountered (the decomp version for `.text` functions) and discards the second (the split version). But this means:

1. **All symbols the linker expects must exist** — if your decomp `.obj` is missing a global variable or vtable that other units reference, you get an unresolved external.
2. **COMDATs must be compatible** — template instantiations, inline functions, and RTTI emitted as COMDAT (`IMAGE_COMDAT_SELECT_ANY`) must be present with correct content. If our compiler emits `IMAGE_COMDAT_SELECT_NODUPLICATES` for something the split `.obj` has as `SELECT_ANY`, the linker warns.
3. **`.pdata` must be well-formed** — the Xbox 360 linker validates `.pdata` entries are sorted by `BeginAddress` ascending, and that entries reference valid `.text` symbols. Malformed `.pdata` causes `LNK1223`.

## What Must Match for a Unit to Be Linkable

### 1. `.text` — Code

**Requirement**: All functions must produce identical machine code, or the differences must be limited to:
- **Register allocation mismatches** — fixable post-build with `scripts/obj_regswap_patcher.py`
- **ICF-merged functions** — these compare against a `merged_<addr>` target that has been folded to another address by the linker. The `.obj`-level code is correct; the mismatch is a linker artifact.

**What breaks linking**: Nothing — `.text` mismatches don't cause link errors, they cause runtime behavioral differences. A function that's 95% matched will link fine but may behave differently.

### 2. `.rdata` — Read-Only Data

**Requirement**: All read-only data symbols must exist with correct content. This includes:

- **Vtables** (`??_7ClassName@@6B@`): The vtable must have the correct number of entries pointing to the correct virtual functions. If a class hierarchy is wrong (missing virtual, wrong override), the vtable will differ.
- **RTTI type descriptors** (`??_R0`, `??_R1`..`??_R4`): Generated by `/GR`. These encode class names, inheritance chains, and virtual base offsets. They match if the class declaration matches.
- **String literals** (`??_C@_0*`): Must match the original strings exactly.
- **Float/double constants** (`__real@3f800000`): Emitted as COMDAT symbols in `.rdata`. Must be present for code that references them.
- **`const` global arrays**: Switch jump tables, lookup tables, etc.

**What breaks linking**: Missing vtable entries cause unresolved externals when other units call virtual functions. Missing RTTI breaks `dynamic_cast`. Missing float constants cause unresolved references from `.text` code.

**Known gap — float constant pooling**: The original compiler sometimes pooled float literals adjacent to static data, using a single `lis` to address both. Our compiler emits separate `.rdata` symbols for each float. This causes a 1-2 instruction difference in `.text` (extra `lis`+`ori` for the address) but doesn't break linking.

### 3. `.data` — Initialized Globals

**Requirement**: All initialized static/global variables must exist with correct initial values and sizes. This includes:
- Static member variables (`ClassName::sInstance`)
- Global state (`gDebug`, `TheDebug`)
- Static locals (mangled names like `?count@?1??Func@@...`)

**What breaks linking**: Missing globals cause unresolved externals. Wrong sizes cause data corruption at runtime (adjacent globals overlap).

### 4. `.bss` — Uninitialized Globals

**Requirement**: All uninitialized globals must exist with correct sizes. The content is zero-initialized at load time.

**What breaks linking**: Same as `.data` — missing symbols or wrong sizes.

### 5. `.pdata` — Exception Tables

**Requirement**: The compiler generates one `IMAGE_CE_RUNTIME_FUNCTION_ENTRY` per function with a stack frame:

```c
typedef struct {
    uint32_t BeginAddress;              // RVA of function start
    uint32_t PrologLen    : 8;          // Prolog length in bytes
    uint32_t FuncLen      : 22;         // Function length in 4-byte units
    uint32_t ThirtyTwoBit : 1;          // Always 1 for PPC
    uint32_t ExceptionFlag: 1;          // 1 if function has try/catch
} IMAGE_CE_RUNTIME_FUNCTION_ENTRY;
```

If `.text` matches, `.pdata` matches — the compiler deterministically generates prolog/function lengths from the same code. Entries must be sorted ascending by `BeginAddress` or the linker rejects the object with `LNK1223`.

For functions with `ExceptionFlag=1`, an 8-byte `PDATA_EH` record sits in `.text` immediately before the function:

```c
struct PDATA_EH {
    void* pHandler;      // → __CxxFrameHandler
    void* pHandlerData;  // → FuncInfo in .rdata
};
```

**What breaks linking**: Malformed `.pdata` causes fatal `LNK1223`. In practice, since the compiler generates `.pdata` from the same `.text`, this isn't a manual concern — if your code compiles, `.pdata` is correct.

### 6. COMDAT Sections

COMDAT sections handle symbols that appear in multiple translation units (templates, inline functions, RTTI). MSVC marks these with a COMDAT selection type:

| Selection | Behavior |
|-----------|----------|
| `IMAGE_COMDAT_SELECT_ANY` | Linker picks any copy, discards duplicates silently |
| `IMAGE_COMDAT_SELECT_NODUPLICATES` | Linker warns on any duplicate (LNK4006) |
| `IMAGE_COMDAT_SELECT_EXACT_MATCH` | Linker verifies all copies are identical |
| `IMAGE_COMDAT_SELECT_LARGEST` | Linker keeps the largest copy |

The MSVC X360 compiler uses `SELECT_NODUPLICATES` for some explicit template specializations, which generates LNK4006 warnings even for legitimate duplicates. This is a known compiler behavior — the 275 remaining LNK4006 warnings from this are considered acceptable noise.

**What breaks linking**: If a COMDAT function is present in the decomp `.obj` but not in the split `.obj` (or vice versa), the linker may resolve cross-unit references to the wrong copy, changing code layout.

### 7. Unwind Thunks (`.text$x`)

`__unwind$FuncName` thunks are compiler-generated cleanup handlers for C++ exception handling. They live in `.text$x` COMDAT sections. The compiler generates these deterministically from the function body — if the function matches, the unwind thunk matches.

## Acceptable Gaps

Not everything needs to be byte-perfect for a unit to be safely linkable:

### Safe to Link Despite Mismatch

| Gap Type | Why It's Safe | How to Verify |
|----------|---------------|---------------|
| **Register allocation** in `.text` | Functionally equivalent code, fixable with `obj_regswap_patcher.py` | Unicorn behavioral comparison shows EQUIVALENT |
| **ICF-merged function** at 0% | The function code is correct in the `.obj`; the linker merged it to another address in the original. Our linker will do the same (or not — either way is functionally correct) | `has_linker_merged = 1` in DB |
| **`.rdata` address differences** | Vtable entries containing absolute VAs differ because our build has different addresses. The structure is correct; only the baked-in pointers differ. | Verify vtable entry count and function order match |
| **Float constant pooling** | 1-2 extra instructions in `.text` to load a float from a different `.rdata` offset. Functionally identical. | Unicorn EQUIVALENT verdict |
| **LNK4006 NODUPLICATES warnings** | MSVC X360 quirk — warns about legitimate COMDAT duplicates. Doesn't affect correctness. | 275 known warnings, documented |

### NOT Safe to Link

| Gap Type | Risk | Symptom |
|----------|------|---------|
| **Missing member variable** | Other units access wrong offsets → data corruption | Unresolved external or silent miscompilation |
| **Wrong class hierarchy** | Vtable layout differs → virtual calls dispatch to wrong functions | Crash at runtime |
| **Missing static variable** | Other units can't find the global | Unresolved external |
| **Wrong function signature** | Caller passes wrong arguments or reads wrong return | Crash or corruption |
| **Logic differences in `.text`** | Function behaves differently | Runtime bugs |

## Practical Workflow for Marking Units as Matching

1. **Check function completion**: All functions in the unit should be COMPLETE or AT_LIMIT in the orchestrator DB.

2. **Verify `.text` match**: Run `objdiff` on the unit. All functions should be 100% or have only register/merged gaps.

3. **Apply register patches**: Run `obj_regswap_patcher.py --batch` after `ninja` to fix register allocation differences.

4. **Check `.rdata`/`.data`/`.pdata`**: objdiff reports section-level match percentages. `.pdata` should be ~100%. `.rdata` may be lower due to address-dependent content — verify the structural correctness (right vtable entries, right RTTI hierarchy) rather than expecting byte-identical content.

5. **Verify no missing symbols**: Build with the unit set to `"Matching"` and check for new unresolved externals or LNK4006 warnings that weren't there before.

6. **Test**: If possible, test the linked XEX in Xenia to verify runtime behavior.
